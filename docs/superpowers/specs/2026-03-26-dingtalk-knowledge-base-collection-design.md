# 钉钉万事通一期知识库收集方案

## 1. 背景与目标

本项目面向公司内部员工，计划在钉钉中建设“万事通”问答能力。一期目标不是建设完整档案中心，也不是一次性沉淀所有制度文档，而是先完成可直接用于问答的知识库收集，为后续知识整理、导入和问答上线打基础。

本阶段仅聚焦第一任务：收集知识库。

一期收集对象为以下 6 个部门：

- 人力行政部
- 销售运营部
- 客服部
- 平台增长部
- 法务部
- 供应链

## 2. 什么是本项目中的“知识库”

本项目中的知识库，指的是适合员工在钉钉万事通中直接提问，并能够得到标准回答或明确指引的内容。

满足以下条件的内容，定义为一期应收集的知识：

- 员工会反复提问
- 部门能提供相对统一的标准答案
- 能追溯到制度、流程、系统规则或负责人
- 能回答“怎么办、找谁、在哪里操作、什么情况例外”

一期优先收集以下 4 类知识：

- FAQ 类：高频咨询问题及标准答案
- 规则类：制度口径、审批规则、权限边界
- 流程类：办理步骤、系统入口、协作路径
- 异常类：特殊情况处理、升级路径、转人工条件

一期不优先收集以下内容：

- 长篇分析报告
- 会议纪要
- 没有结论的讨论材料
- 仅有附件、没有文字说明的资料
- 过期版本制度
- 无明确适用场景的数据报表

## 3. 收集原则

为保证后续能顺利做成问答，知识库收集需遵循以下原则：

- 先高频，后长尾
- 先标准答案，后复杂判断
- 先问答内容，后原始文档
- 先结构化收集，后补充附件
- 每条知识必须有负责人和依据

项目目标不是“收文档”，而是“沉淀可问答的知识条目”。

## 4. 一期推荐收集策略

推荐采用“先做高频问答，再补制度原文”的方式推进。

### 4.1 推荐方案

第一轮只要求各部门提交：

- 前 30 到 50 个高频问题
- 每个问题的标准答案
- 对应制度、流程或系统依据
- 特殊情况说明
- 转人工或升级条件
- 内容负责人

### 4.2 推荐原因

- 最适合钉钉万事通的问答场景
- 启动快，容易产出一期样板
- 能避免一开始收集大量不能直接用于问答的资料
- 有利于项目组统一整理、统一口径

## 5. 整体推进计划

由于当前以线上沟通为主，建议采用“异步回收 + 小范围澄清”的方式推进。

### 5.1 阶段一：统一标准

由项目组先统一以下内容：

- 知识库定义和收集范围
- 提交模板
- 收集优先级
- 各部门对接窗口
- 时间安排和反馈机制

本阶段产出物：

- 一份知识库收集说明
- 一份统一模板
- 一份部门对接清单

### 5.2 阶段二：线上发起与异步回收

向各部门窗口人发起收集说明，明确：

- 本次只收适合内部问答的知识，不做全量文档归档
- 每个部门先提交前 30 到 50 个高频问题
- 每条问题尽量补充标准答案、依据文件、例外情况和负责人
- 优先收集高频、易错、跨部门高咨询量问题

本阶段产出物：

- 各部门第一版问题清单
- 初步依据文件目录
- 维护人名单

### 5.3 阶段三：项目组整理与二次回收

项目组对部门回收内容进行结构化整理，统一转换成问答条目，并识别以下问题：

- 提问方式不自然
- 答案过于笼统
- 缺少依据文件
- 缺少例外说明
- 缺少转人工边界
- 有多个版本口径冲突

之后将整理后的版本发回部门确认。

本阶段产出物：

- 各部门结构化问答表
- 冲突问题清单
- 待确认问题清单

### 5.4 阶段四：线上澄清与验收

对于异步无法确认的问题，再分别约 20 到 30 分钟线上沟通确认，不需要一开始就开长会。

重点确认：

- 标准答案是否准确
- 是否适合在万事通中对全员开放
- 是否存在敏感边界
- 哪些问题必须转人工
- 谁负责后续更新

本阶段产出物：

- 各部门一期确认版知识条目
- 不适合入库内容清单
- 后续更新责任人清单

## 6. 线上收集的实际执行方式

## 6.1 对接角色

每个部门至少确认以下 3 类角色：

- 部门负责人：确认范围、优先级、内容归属
- 业务骨干：提供高频问题和实际处理方式
- 资料或系统负责人：提供制度、流程、系统入口、附件位置

## 6.2 线上推进方式

建议按以下顺序执行：

1. 向部门负责人或窗口人发起说明
2. 发送统一模板，要求先提交第一版
3. 项目组跟进答疑和催收
4. 项目组先整理，再回给部门确认
5. 对歧义问题单独约短会澄清

## 6.3 对外沟通建议话术

建议使用以下表达方式：

“本次我们在为钉钉万事通整理内部知识库，一期仅收集适合员工问答的内容，不做全量档案归集。请优先整理本部门高频问题、标准答案、流程指引和例外处理，首轮先提供 30 到 50 个最常见问题即可。项目组会负责统一整理，你们主要协助确认口径和依据。”

## 6.4 催收与质量控制

线上推进中最容易出现的问题包括：

- 直接提交大量原始文档，不按模板填写
- 只写制度标题，不写员工会问的问题
- 只给原则，没有操作步骤
- 没写特殊情况和边界
- 没有内容负责人

建议控制方式：

- 发出模板后 2 天内提醒一次
- 不要求一次性填完整，先收第一版
- 项目组帮忙把原始内容翻译成问答形式
- 对模糊内容集中整理后再回部门确认

## 7. 统一收集模板建议

建议所有部门统一按下列字段提交，便于后续导入和知识治理。

| 字段 | 说明 |
| --- | --- |
| 所属部门 | 知识归属部门 |
| 问题编号 | 用于管理和追踪 |
| 问题分类 | 如制度、流程、系统、异常、审批等 |
| 标准问题 | 员工最可能直接提的问题 |
| 相似问法 | 同义表达、口语化问法 |
| 标准答案 | 可以直接用于问答的统一回答 |
| 适用范围 | 哪些人群、哪些场景适用 |
| 处理步骤 | 如需办理，写清操作步骤 |
| 例外情况 | 特殊情形说明 |
| 转人工条件 | 何时不能自动答复，需转人工 |
| 依据文件 | 对应制度、流程、SOP 名称 |
| 附件或链接 | 文档、系统、表单入口 |
| 内容负责人 | 负责确认内容的人 |
| 更新时间 | 用于判断是否过期 |
| 备注 | 其他补充说明 |

## 8. 各部门应收集的知识库范围

## 8.1 人力行政部

优先收集：

- 入职、转正、调岗、离职流程
- 请假、考勤、加班、补卡规则
- 出差、报销、福利、社保公积金说明
- 工牌、门禁、工位、办公用品、固定资产申请
- 用章、访客、会议室、车辆等行政流程
- 员工证明、档案、合同相关常见问题
- OA 或人事系统常见操作说明
- 审批路径和联系人信息

典型问题示例：

- 请假怎么申请
- 补卡流程是什么
- 离职需要提前多久提
- 社保从什么时候开始缴纳
- 办公电脑坏了找谁

## 8.2 销售运营部

优先收集：

- 线索分配规则、客户归属规则
- 折扣、报价、审批权限说明
- CRM 字段口径、录入规范、商机阶段定义
- 合同、签约、回款、续费流程
- 业绩统计口径和归属规则
- 销售政策、返利政策、活动政策
- 提报入口、表单和审批路径
- 与法务、财务、客服的协作流程

典型问题示例：

- 客户归属怎么判定
- 这个折扣需要谁审批
- 业绩按签约还是按回款计算
- CRM 里这个字段怎么填
- 合同提交后下一步是什么

## 8.3 客服部

优先收集：

- 高频用户咨询及标准回复
- 工单分类、标签规则、优先级定义
- 投诉、退款、售后、异常处理流程
- 身份核验规则和敏感信息处理规范
- 常见系统问题的标准排查口径
- 升级给技术、产品、法务、销售的场景
- 服务时效标准和响应规则
- 客服系统或工单系统操作说明

典型问题示例：

- 客户要求退款怎么处理
- 什么情况需要升级投诉
- 工单标签怎么选
- 账号异常需要核验哪些信息
- 这个问题该转技术还是产品

## 8.4 平台增长部

优先收集：

- 增长相关核心指标定义
- 渠道、流量来源和归因口径
- 活动立项、上线、复盘流程
- 用户分层规则和触达规则
- A/B 测试基础流程和术语解释
- 数据看板和报表口径说明
- 与产品、设计、技术、运营的协作流程
- 当前稳定执行的规则和常见问题

典型问题示例：

- 新增用户的定义是什么
- 这个活动上线前需要谁确认
- 渠道数据看哪个报表
- 自然流量和投放流量怎么区分
- A/B 测试申请流程是什么

## 8.5 法务部

优先收集：

- 合同审核提交流程
- 常用合同模板的适用场景
- 哪些条款可改，哪些必须升级审核
- 用章、授权、证照使用流程
- 宣传文案、合作协议、隐私条款等合规边界
- 常见法律咨询的标准口径
- 纠纷、投诉、风险事件上报流程
- 法务必须介入的触发条件

典型问题示例：

- 合同审核怎么提
- 这个合同是否可以直接套模板
- 哪些条款修改后必须找法务
- 宣传文案发布前是否需要审核
- 盖章申请需要什么材料

## 8.6 供应链

优先收集：

- 采购申请、下单、到货、验收、付款流程
- 供应商准入、变更、考核规则
- 库存查询、盘点、预警口径
- 发货、退货、换货流程
- 缺货、延迟、错发、破损等异常处理规则
- 采购、仓储、物流系统操作说明
- 单据填写要求和审批路径
- 与财务、业务、仓储、物流的协作节点

典型问题示例：

- 采购申请提给谁
- 收货后怎么做验收
- 缺货时怎么处理
- 退货流程怎么走
- 新增供应商需要哪些资料

## 9. 收集优先级建议

建议按以下顺序推进：

第一批优先部门：

- 人力行政部
- 客服部
- 法务部

原因：

- 员工咨询频率高
- 规则相对标准
- 更容易快速形成一期样板

第二批推进部门：

- 销售运营部
- 供应链
- 平台增长部

## 10. 一期验收标准

一期收集完成后，建议按以下标准验收：

- 是否能够被员工直接提问
- 是否有明确、稳定的标准答案
- 是否能追溯到依据文件或负责人
- 是否写明了例外情况
- 是否写明了转人工边界
- 是否适合对公司内部员工开放

## 11. 建议的一期输出物

一期收集工作完成后，建议至少形成以下成果：

- 一份统一的知识库收集模板
- 6 个部门的高频问答条目清单
- 各部门依据文件目录
- 不适合直接入库的问题清单
- 内容负责人和后续维护责任人清单

## 12. 下一步建议

在本方案确认后，建议立即执行以下动作：

1. 建立统一收集表
2. 明确 6 个部门窗口人
3. 发出线上收集说明
4. 优先回收第一批 3 个部门内容
5. 由项目组完成第一轮结构化整理
6. 组织短会澄清歧义问题
7. 形成一期可入库知识条目
