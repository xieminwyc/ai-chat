# AI Chat 账号体系、游客试用与历史合并设计

## 文档概览

| 项目 | 内容 |
| --- | --- |
| 文档主题 | AI Chat 账号体系、游客试用与历史合并设计 |
| 当前阶段 | 第一阶段产品设计 |
| 面向对象 | 项目 owner、后续实施者、未来回看身份体系的人 |
| 推荐方向 | `邮箱 + 密码`、必须验证邮箱、保留游客试用、注册后可选择合并游客历史 |
| 目标仓库 | `AI Chat` |

## 背景

当前项目已经具备基础聊天能力：

- 首页聊天界面由 `src/components/chat-app.tsx` 驱动
- 服务端聊天接口由 `src/app/api/chat/route.ts` 提供
- 数据库当前只有 `Chat` 和 `Message`

现状问题也很明确：

1. 聊天记录没有用户维度，天然是全局共享思路。
2. 没有登录注册，无法把会话稳定归属到某个用户。
3. 无法支持“跨设备继续聊”“保存个人历史”“未来做额度、会员、知识库”等能力。
4. 如果直接强制登录后才能使用，会降低早期产品体验和转化。

因此，第一阶段需要先补齐最小可用的身份体系，同时保留低门槛试用路径。

## 第一阶段目标

第一阶段只解决这几件事：

1. 支持 `邮箱 + 密码` 注册与登录。
2. 用户必须完成邮箱验证后，才能正常继续聊天。
3. 保留游客模式，但游客有试用次数限制。
4. 游客聊天记录与额度状态在同一浏览器内可延续，刷新不会重置。
5. 注册并完成邮箱验证后，用户可以选择是否合并当前游客历史。
6. 所有聊天记录都需要明确归属到某个正式用户或某个游客身份。

## 第一阶段不做

- 第三方登录
- 邮箱魔法链接登录
- 忘记密码
- 修改密码
- 多设备安全管理
- 跨浏览器游客同步
- 复杂风控与封禁系统
- 账号设置页的大而全能力

## 方案比较

### 方案 A：游客与正式账号共存，我方推荐

核心思路：

- 保留游客体验，允许先试聊
- 游客有明确试用上限
- 游客历史保存在服务端
- 正式用户通过邮箱 + 密码注册，完成邮箱验证后激活
- 激活后询问是否合并当前游客历史

优点：

- 产品路径顺滑，先体验再转化
- 能真正解决“聊天按用户隔离”
- 对当前代码改动可控，不需要推翻聊天主链路
- 方便后续扩展会员、知识库、额度、跨设备同步

缺点：

- 需要引入游客身份模型
- 需要处理游客转正时的数据迁移

### 方案 B：游客只存本地，注册后才真正入库

优点：

- 上线更快
- 后端改动较少

缺点：

- 游客换浏览器或清缓存即丢失
- 合并历史会更脆
- 未来做额度、分析、风险控制都不顺

### 方案 C：第一版直接做完整账号平台

优点：

- 架构完整
- 后续返工少

缺点：

- 第一阶段明显过重
- 容易把项目从产品验证拖成基础设施工程

### 推荐结论

采用 `方案 A`。

原因：

- 能在当前阶段同时兼顾体验、转化和后续扩展
- 与现有 Next.js + Prisma 聊天项目最兼容
- 能把“聊天记录归属”这个核心问题一次解决

## 产品规则

### 游客模式

- 游客无需登录即可开始聊天
- 游客有固定试用次数上限，例如 `3` 次或 `5` 次
- 游客聊天记录保存在服务端
- 游客刷新页面后：
  - 仍能看到之前的聊天记录
  - 试用状态不会重置
  - 如果额度已用完，仍不可继续发送消息
- 游客关闭页面后再次进入：
  - 在同一浏览器内，恢复同一游客身份
  - 历史与额度状态继续沿用
- 换浏览器后：
  - 视为新游客
  - 原游客历史不会自动同步

### 注册与验证

- 注册方式为 `邮箱 + 密码`
- 用户注册后，系统立即发送验证邮件
- 账号在验证前处于“未验证邮箱”状态
- 未验证邮箱账号不能继续完整聊天
- 用户点击邮件验证链接后，服务端只负责把账号标记为“已验证”
- 邮件验证成功不应直接触发游客历史合并

这样设计是为了避免跨设备验证带来的错位问题：

- 用户可能在设备 A 以游客身份聊天
- 却在设备 B 点开验证邮件
- 设备 B 没有设备 A 的游客 cookie，无法安全识别原游客身份

因此第一阶段要明确解耦：

- `邮箱验证` 负责激活账号
- `游客历史合并` 只在带有原游客 cookie 的浏览器中发生

### 游客历史合并

- 用户在完成邮箱验证后，如果回到原游客浏览器并且当前处于已登录状态，前端再检测是否存在未合并的本地游客身份
- 只有在“当前浏览器既有已验证登录态，又有未合并 guestSession”时，才弹出合并提示：
  - 合并当前游客聊天记录到我的账号
  - 不合并，作为新账号开始
- 如果选择合并：
  - 当前游客名下的所有聊天归属转移到用户
  - 游客身份标记为已合并，避免重复迁移
- 如果选择不合并：
  - 用户账号从空白状态开始
  - 游客历史继续留在该游客身份下，按后续生命周期策略自然存在或过期

## 数据模型设计

### 新增模型

#### `User`

建议字段：

- `id`
- `email`
- `passwordHash`
- `emailVerifiedAt`
- `createdAt`
- `updatedAt`

职责：

- 表示正式账号主体
- 作为正式聊天记录、后续配额、知识库等能力的拥有者
- 删除用户时，建议级联删除其名下聊天记录

#### `GuestSession`

建议字段：

- `id`
- `guestToken`
- `trialMessageCount`
- `mergedAt`
- `expiresAt`
- `createdAt`
- `updatedAt`

职责：

- 表示游客身份
- 记录游客试用计数
- 表示某个浏览器内的临时聊天主体
- 如果未来需要清理过期游客数据，可通过删除 `GuestSession` 级联清理其聊天记录

### 修改现有模型

#### `Chat`

建议新增：

- `userId`，可空
- `guestSessionId`，可空

约束规则：

- 每条 `Chat` 必须且只能归属到一个主体
- 主体要么是 `User`
- 要么是 `GuestSession`
- 不允许两边都为空
- 不允许两边同时有值

Prisma schema 本身无法直接优雅表达这个 XOR 约束，因此第一阶段建议双层兜底：

1. 应用层在创建与迁移 `Chat` 时强制校验归属
2. 数据库 migration 中手动加入 `CHECK` 约束，例如：

```sql
ALTER TABLE "Chat" ADD CONSTRAINT "chat_owner_check"
CHECK (
  ("userId" IS NOT NULL AND "guestSessionId" IS NULL) OR
  ("userId" IS NULL AND "guestSessionId" IS NOT NULL)
);
```

#### `Message`

`Message` 继续从属于 `Chat`，主结构不变。

### 关系说明

```text
User 1 --- n Chat 1 --- n Message
GuestSession 1 --- n Chat 1 --- n Message
```

这样做的好处是：

- 不推翻现有 `Chat / Message` 主链路
- 游客和正式用户共用同一套聊天系统
- 历史合并本质上只是把 `Chat` 的归属从 `GuestSession` 改到 `User`

## 身份识别设计

### 游客身份

- 初次进入网站时，系统创建一个 `guestSession`
- 浏览器保存一个稳定的游客标识
- 后续请求通过该标识恢复游客身份

第一阶段建议通过以下方式实现：

- `httpOnly cookie` 保存游客 token，优先推荐
- 或使用浏览器本地 token 配合接口传递

推荐优先使用 cookie，原因：

- 更贴近服务端身份识别
- 比单纯依赖 localStorage 更适合后续与登录态并存
- 也能避免前端伪造游客身份

### 正式用户身份

- 注册成功后建立用户记录
- 登录成功后建立正式会话
- 后续聊天接口优先识别正式用户身份

## 用户流程设计

### 流程一：游客试用

```text
进入首页
-> 自动创建或恢复 guestSession
-> 游客发送消息
-> 服务端检查当前 trialMessageCount
-> 未超限则允许发送并记账
-> 超限则阻止发送并引导注册
```

### 流程二：注册

```text
游客点击注册
-> 填写邮箱与密码
-> 创建用户
-> 生成邮箱验证 token
-> 发送验证邮件
-> 页面进入“等待验证”状态
```

### 流程三：邮箱验证

```text
用户点击验证链接
-> 服务端校验 token
-> 写入 emailVerifiedAt
-> 标记账号已验证
-> 返回“验证成功，请回到原设备刷新或登录”的结果页
```

### 流程四：合并游客历史

```text
用户回到原游客浏览器
-> 已登录且已验证
-> 前端检测当前浏览器存在未合并 guestSession
-> 弹出“是否合并游客历史”
-> 用户选择“合并”
-> 校验当前 guestSession 未被合并
-> 将该 guestSession 下所有 Chat 转移到 User
-> 标记 guestSession 已合并
-> 进入正常已登录聊天状态
```

### 流程五：不合并历史

```text
用户选择“不合并”
-> 保持 guestSession 历史不动
-> 用户账号聊天列表为空
-> 进入正常已登录聊天状态
```

## 前端设计

第一阶段不推翻现有聊天页结构，而是在聊天界面外侧补一层“身份壳”。

### 新增前端状态

- 当前身份：游客 / 已登录未验证 / 已验证用户
- 当前游客剩余试用次数
- 当前是否存在待合并的游客历史
- 注册弹窗
- 登录弹窗
- 等待邮箱验证提示
- 重发验证邮件入口
- 原设备登录后触发的历史合并选择界面

### 关键界面节点

#### 游客可用阶段

- 正常显示聊天界面
- 在接近试用上限时，提前提示注册价值

#### 游客额度耗尽阶段

- 历史可读
- 输入框禁用或发送行为被阻止
- 明确提供注册入口

#### 未验证邮箱阶段

- 显示“请前往邮箱完成验证”
- 提供“重发验证邮件”
- 暂不允许继续完整聊天

#### 验证成功后

- 如果当前浏览器有未合并 guestSession，再弹出“是否合并当前游客聊天记录”
- 如果当前浏览器没有 guestSession，只提示验证成功并进入正常账号状态

## 服务端接口设计

### 继续保留并改造现有接口

#### `GET /api/chat`

职责：

- 只返回当前主体自己的聊天列表或聊天消息

规则：

- 已登录用户只看到自己的聊天
- 游客只看到自己的聊天
- 不再存在“全局共享聊天列表”

#### `POST /api/chat`

职责：

- 发送一条新消息并获取助手回复

规则：

- 先识别当前身份
- 如果是游客，先检查试用额度
- 如果是未验证邮箱用户，阻止继续正常聊天
- 新建 Chat 时要挂到正确主体下

#### `PATCH /api/chat`

职责：

- 修改当前会话标题

规则：

- 只能修改自己名下的 Chat

#### `DELETE /api/chat`

职责：

- 删除当前会话

规则：

- 只能删除自己名下的 Chat

### 新增接口

#### `GET /api/auth/session` 或 `GET /api/me`

职责：

- 为前端提供当前身份与可用状态

返回建议包括：

- 当前是否已登录
- 当前是否是游客
- 当前用户是否已验证邮箱
- 游客已使用次数与试用上限
- 当前浏览器是否存在待合并游客历史

引入这个接口的原因：

- 游客与登录态推荐保存在 `httpOnly cookie`
- 前端 JavaScript 不能直接读取这些 cookie
- `chat-app.tsx` 需要依赖服务端状态决定界面展示与拦截逻辑

#### `POST /api/auth/register`

职责：

- 注册用户
- 创建邮箱验证 token
- 发送验证邮件

#### `POST /api/auth/login`

职责：

- 账号登录

#### `POST /api/auth/logout`

职责：

- 退出登录

#### `POST /api/auth/verify-email`

职责：

- 消费邮箱验证 token
- 激活账号

#### `POST /api/auth/resend-verification`

职责：

- 重发验证邮件

#### `POST /api/guest/merge`

职责：

- 将当前游客历史合并进当前已验证用户

## 权限与校验设计

建议把身份与权限判断抽成统一逻辑，而不是散落在各个接口中。

所有聊天相关请求都先经过这三步：

1. 识别当前请求主体是谁
2. 判断目标 Chat 是否归属该主体
3. 判断该主体是否有资格继续执行本次操作

### 权限规则

- 游客：
  - 可在试用额度内聊天
  - 只能访问自己名下聊天
- 已登录但未验证邮箱用户：
  - 可完成验证相关操作
  - 不可继续完整聊天
- 已验证用户：
  - 可完整使用聊天能力
  - 只能访问自己名下聊天

## 邮箱验证设计

第一阶段采用最简单可靠的 token 方案。

### 核心流程

- 注册成功后生成一次性验证 token
- token 存数据库并关联用户
- 邮件中包含验证链接
- 用户点击链接后，服务端校验 token 是否存在、是否过期、是否已使用
- 校验通过后写入 `emailVerifiedAt`

### 第一阶段不做

- 邮箱更换
- 邮件模板后台管理
- 多通道邮件供应商切换

## 游客试用与生命周期设计

### 试用次数

- 试用次数按 `GuestSession` 计数
- 不按页面内存计数
- 不按单次页面生命周期计数
- 计数更新必须考虑并发场景，不能使用“先查再加”的脆弱写法

这样能保证：

- 刷新页面不会重置试用
- 试用规则不容易被简单绕过

实现上建议使用数据库事务或 Prisma 原子更新能力，例如：

- 对 `trialMessageCount` 使用 `increment`
- 在同一事务中完成“检查是否超限”和“记账”

如果只做“读取当前计数 -> 判断 -> 再单独 update”，在双标签页或高频点击时可能突破上限

### 历史保留

第一阶段建议：

- 同一浏览器内恢复历史
- 不做跨浏览器游客同步
- 不在界面上强提示“游客记录只保存在当前浏览器”

### 生命周期

第一阶段可以先允许游客记录自然保留，后续再补清理策略。

如果后面需要成本控制，可再加：

- 7 天或 30 天后清理长期未使用的 `GuestSession`

## 测试重点

第一阶段最值得覆盖的测试包括：

1. 游客只能读取自己的聊天记录。
2. 正式用户只能读取自己的聊天记录。
3. 游客超出试用额度后不能继续发送消息。
4. 游客刷新页面后仍能看到历史，但额度不会重置。
5. 未验证邮箱账号不能继续完整聊天。
6. 验证成功后可以选择合并或不合并游客历史。
7. 合并后聊天归属正确，且不会重复合并。
8. 不合并时，账号历史为空，游客历史保持不动。
9. 用户不能重命名或删除不属于自己的聊天。

## 实施建议

推荐分两步推进：

### 第一步：先把数据归属和身份识别做对

- 引入 `User` 与 `GuestSession`
- 给 `Chat` 补归属字段
- 调整所有聊天查询与写入逻辑

### 第二步：再补注册、验证、合并流程

- 注册登录接口
- 邮箱验证链路
- 游客额度与界面提示
- 合并历史流程

这样实施时更稳，也更容易定位问题。

## 认证实现选型

第一阶段不建议手写完整 JWT 登录体系。

推荐结论：

- 优先使用 `iron-session` 管理 `httpOnly cookie` 会话
- 使用 `bcryptjs` 处理密码哈希
- 邮箱验证 token 独立存库并单独校验

推荐原因：

- 和当前基于 Route Handlers 的项目结构更贴合
- 足够覆盖第一阶段的账号、游客、登录态需求
- 比手写 JWT 与 cookie 细节更稳
- 复杂度明显低于一次性引入完整 Auth 平台

关于 `Auth.js`：

- 如果项目短期内就要接第三方登录，`Auth.js` 会更有吸引力
- 但对当前“邮箱密码 + 游客试用 + 历史合并”的第一阶段目标，`iron-session` 更轻、更直接

## 最终结论

第一阶段推荐落地为：

- `邮箱 + 密码` 注册登录
- 必须验证邮箱后才能正常使用
- 保留游客模式
- 游客可试用，但有明确消息次数上限
- 游客历史在同一浏览器内保留，刷新不丢失、不重置额度
- 换浏览器视为新游客
- 验证成功后，用户可选择是否合并当前游客历史

这套设计能在不推翻现有聊天系统的前提下，补齐身份基础设施，并为未来扩展：

- 通用聊天工具
- 学习 / 编程陪练
- 个人知识库
- 写作与灵感整理

留出清晰空间。
